## Bug #1: 环切模式收缩步长计算错误

### 问题描述
在 `mock_cam.cpp` 的 `GenerateSurfaceToolpath` 函数中，环切模式使用 `min_range` 作为收缩基准：

```cpp
double min_range = std::min(u_range, v_range);
double shrink_step = min_range / (2.0 * (num_paths + 1));
```

当UV参数范围差异很大时（如 U=[0,1], V=[0,80]），会导致：
- U方向（范围小）收缩过快，可能提前到零
- V方向（范围大）收缩太慢，浪费空间

### 修复方案
分别计算U和V方向的收缩步长：

```cpp
double shrink_step_u = u_range / (2.0 * (num_paths + 1));
double shrink_step_v = v_range / (2.0 * (num_paths + 1));

// 在循环中使用
double shrink_u = layer * shrink_step_u;
double shrink_v = layer * shrink_step_v;
```

### 验证结果
✓ 成功生成 43,192 个刀轨点，点分布合理

## Bug #2: 行切模式边界重复点

### 问题描述
在行切模式中，边界处理使用了条件判断：

```cpp
double u = u_min + j * step_u;
if (u > u_max) u = u_max;
```

这会导致当计算值超过边界时被强制设为边界值，可能产生重复点。

### 修复方案
使用 `std::min` 直接限制：

```cpp
double u = std::min(u_min + j * step_u, u_max);
double v = std::min(v_min + i * step_v, v_max);
```

### 验证结果
✓ 生成 441 个刀轨点，无重复点

## Bug #3: Python客户端缺少错误处理

### 问题描述
`python_client/client.py` 中没有捕获 gRPC 连接和调用错误，导致异常时程序直接崩溃。

### 修复方案
添加 try-except 错误处理：

```python
try:
    channel = grpc.insecure_channel('localhost:50051')
    stub = CamCalculationServiceStub(channel)
except Exception as e:
    print(f"Error: Failed to create gRPC channel: {e}")
    return

# 在调用时也添加错误处理
try:
    shape, largest_face, points = calculate_surface_toolpath(...)
except grpc.RpcError as e:
    print(f"gRPC Error: {e.code()} - {e.details()}")
    return
```

### 验证结果
✓ 正确捕获并显示 gRPC 错误信息
